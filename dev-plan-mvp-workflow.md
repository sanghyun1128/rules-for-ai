## 0. Project Sources of Truth

### 0.1 Authority Order
작업 중 기준 우선순위는 아래 순서를 따른다.
1. 구현 범위, 공개 API, 타입, 디렉터리 구조, 수용 기준: `PLAN.md`
2. 진행 상태와 작업 보드: `[이슈 트래커]`
3. UI 레이아웃 및 기능 흐름 참고: `[UI 디자인 툴]`

`PLAN.md`가 존재하지 않으면 구현을 시작하지 않는다. 먼저 `PLAN.md`를 만들거나 갱신한 뒤 그 내용을 기준으로 작업한다.

### 0.2 UI Reference (`[UI 디자인 툴]`)
- UI 관련 작업은 구현 전 해당 화면의 전반적인 레이아웃과 흐름을 파악하기 위해 확인한다.
- MVP 단계이므로 픽셀 퍼펙트(Pixel-perfect)를 요구하지 않으며, 핵심 구조와 기능 요소가 존재하고 육안상 비슷한 수준이면 코드로 반영한다.
- 폰트 크기, 미세한 여백, 정확한 색상 코드 등 세부적인 시각적 차이로 인해 작업을 멈추지 않는다.
- 단, 디자인의 요소가 `PLAN.md`의 구현 범위를 완전히 벗어나는 새로운 기능(예: 계획에 없던 복잡한 애니메이션이나 추가 페이지)을 요구할 경우에만 구현을 멈추고 `PLAN.md`를 갱신한다.

### 0.3 Progress Mirror (`[이슈 트래커]`)
- `PLAN.md`와 `[이슈 트래커]`의 동기화 방향은 항상 단방향이다.
1. `PLAN.md`가 기준이다.
2. `[이슈 트래커]`는 `PLAN.md`의 진행 상태를 반영한다.
3. 양측이 충돌하면 `[이슈 트래커]`를 `PLAN.md`에 맞춘다.
4. `[이슈 트래커]`에만 존재하는 작업은 구현 근거가 아니다.

### 0.4 PLAN.md <-> Issue Tracker Sync Contract
- `PLAN.md`의 실행 단계 하나는 `[이슈 트래커]` 작업 하나와 반드시 1:1로 대응한다.
- 트래커의 `작업 이름`은 `PLAN.md` 단계 제목에서 선행 접두어(예: `Step X: `)를 제거한 제목으로 맞춘다.
- `PLAN.md`에 단계가 추가되거나 이름이 바뀌면 대응하는 트래커 작업도 동기화한다.
- 트래커의 `설명(Description)` 필드는 PR 생성 전까지 비워두며, PR 생성 후 해당 PR 링크만 기록한다.
- 트래커 갱신 시점:
  1. `PLAN.md`에 단계가 새로 추가되었을 때
  2. 단계를 실제로 시작할 때 (상태: `진행 중`)
  3. PR이 생성되었을 때 (설명에 링크 추가)
  4. 단계가 머지되어 완료되었을 때 (상태: `완료`, 완료일 기록)

## 1. PLAN.md as Single Source of Truth
- 모든 구현 범위는 반드시 `PLAN.md`를 따른다.
- `PLAN.md`에 정의되지 않은 기능, 리팩터링, 구조 변경은 구현하지 않는다.
- 모호할 때는 아래 섹션 우선순위를 따른다.
  1. Confirmed Decisions
  2. Scope (In / Out)
  3. Public API / Types
  4. Acceptance Criteria
  5. Implementation Steps
- 계획 밖의 변경이 필요하면 `PLAN.md` 수정 -> 트래커 갱신 -> 사용자 확인 -> 구현 재개 순서를 따른다.

---

## 2. HARD STOP RULES (Mandatory)
아래 조건 중 하나라도 발생하면 구현을 즉시 멈춘다.
1. `PLAN.md`가 없거나 현재 구현 대상 단계가 정의되지 않음.
2. 구현에 `PLAN.md`에 없는 변경이 필요함.
3. 한 번에 여러 구현 단계를 동시에 진행하려고 함.
4. 동작이 `Scope (Out)` 정의와 충돌함.
5. 현재 단계와 1:1로 대응하는 트래커 작업이 없음.
6. 독립적으로 검증 가능한 기능 단위를 2개 이상 쌓아 두고 마지막에 한 번에 커밋하려고 함.

**위반 시 행동 지침:** 즉시 멈춤 -> 충돌 규칙 설명 -> `PLAN.md` 수정안 제안 -> 트래커 갱신 -> 확인 대기 -> 재개.

---

## 3. Work Start Procedure (Mandatory)
1. 루트의 `PLAN.md` 존재 여부를 확인한다.
2. `Implementation Steps`에서 정확히 하나의 단계만 선택한다.
3. 대응하는 `[이슈 트래커]` 작업을 찾아 상태를 `진행 중`으로 갱신한다.
4. 해당 단계에 필요한 `PLAN.md` 섹션만 검토한다 (라우팅, 타입, 정책 등).
5. UI 영향이 있는 단계면 구현 전에 `[UI 디자인 툴]`의 관련 화면 레이아웃을 참고한다.
6. `Out (v1 excluded)`에 있는 내용은 절대 구현하지 않는다.

## 4. Branch & Commit Rules
### Branch Rules
- 각 구현 단계 시작 시 `main`에서 새 브랜치를 만든다.
- 브랜치 이름은 단계 번호가 아닌 변경 의도를 설명해야 한다. (예: `feat/feature-slug`, `fix/feature-slug`, `refactor/...`, `chore/...`)
- 하나의 브랜치는 하나의 구현 단계만 담당하며 머지 후 삭제한다.

### Commit Rules
- 각 커밋 시점마다 빌드 가능 상태를 유지한다.
- 커밋 가능한 기능 단위(독립적으로 설명 가능하고 검증 가능한 최소 단위)가 완료되면 즉시 커밋한다.
- "Step 하나 끝나면 마지막에 정리해서 커밋" 방식은 절대 금지한다.
- 커밋 전에 최소한 해당 기능 단위에 필요한 검증 결과를 확보한다.

---

## 5. Code Validation & PR Rules
PR 생성 전 아래를 모두 만족해야 한다.
- Build, Lint, Tests 통과
- 사용하지 않는 코드 제거
- UI 변경 시 `[UI 디자인 툴]`의 핵심 레이아웃 및 기능 요소가 유사하게 반영되었는지 육안 확인 완료

### Pull Request Rules
- PR 제목 형식: `[type] short description` (예: `[feat] initialize project`)
- PR 생성 직후 작업:
  1. 트래커 작업 `설명`에 PR 링크 기록.
  2. 같은 브랜치에서 현재 단계의 `PLAN.md` 체크박스를 `[x]`로 바꾸는 후속 커밋 작성 후 push.

---

## 6. Next Step Condition (Post-Merge)
다음 단계는 현재 PR이 머지된 후에만 시작한다. 사용자가 머지 완료를 알리면 아래의 Post-merge closeout을 자동 수행한다.
1. `main` 브랜치로 checkout 및 최신 내용 pull.
2. `PLAN.md` 체크 상태 반영 확인.
3. 트래커 작업 상태를 `완료`로 변경 및 완료일 기록.
4. 로컬/원격 구현 브랜치 삭제.
5. `main` 워크트리가 깨끗한지 확인 후 사용자에게 결과 보고.
위 과정 중 실패가 발생하면 다음 단계로 넘어가지 않고 즉시 보고한다.
